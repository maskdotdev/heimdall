# 0 Core Contracts and Shared Types — Implementation Specification

## Purpose

This document defines the implementation plan for the `@repo/contracts` package: the shared type and schema layer for a code review system.

This package is the foundation for the entire product. The API server, workers, indexer, importer, retrieval engine, review engine, LLM gateway, publisher, dashboard, and internal tools should all use these contracts instead of inventing local shapes.

The main goal is to make every important object in the system explicit, validated, versioned, and stable.

## Executive summary

Build `@repo/contracts` as a small, dependency-light TypeScript package that exports:

- Runtime schemas.
- Static TypeScript types.
- Validators.
- Common enums.
- Job payload contracts.
- API request/response contracts.
- Public re-exports of index artifact contracts owned by `@repo/index-schema`.
- Review/finding contracts.
- Error and result types.
- Test fixtures.

Recommended schema system:

- Use **TypeBox** as the canonical schema representation because it works naturally with Elysia, produces JSON Schema, and is suitable for high-performance validation.
- Use **Ajv** or TypeBox's compiled validators for runtime validation at boundaries.
- If a downstream library prefers Zod, especially for LLM structured outputs, add a small adapter in the LLM gateway. Do not make Zod the only canonical schema unless you intentionally choose Zod across the whole stack.

Recommended package name:

```text
/packages/contracts
```

Suggested import name:

```ts
import { PullRequestSnapshotSchema, type PullRequestSnapshot } from "@repo/contracts";
```

The most important rule:

> Every boundary object should have both a runtime schema and a TypeScript type.

Boundary objects include webhook-normalized events, job payloads, index artifacts, PR snapshots, context bundles, findings, API DTOs, and usage events.

## Design principles

### 1. Contracts are product infrastructure

The contracts package is not a random collection of types. It is the canonical language of the system.

Every subsystem should speak these shapes:

```text
GitHub adapter
  -> PullRequestSnapshot

Indexer
  -> IndexArtifact

Importer
  -> CodeIndexVersion

Retriever
  -> ContextBundle

Review engine
  -> CandidateFinding[]

Validator
  -> ValidatedFinding[]

Publisher
  -> PublishedFinding[]

Feedback processor
  -> FindingOutcome
```

### 2. Runtime validation matters

TypeScript types disappear at runtime. You need runtime schemas for:

- Webhook payload normalization.
- API requests.
- Queue jobs.
- Index artifact imports.
- LLM structured outputs.
- Internal tool input.
- Test fixture validation.
- Cross-service communication.

Every public contract should have:

```ts
export const SomeSchema = Type.Object({ ... });
export type Some = Static<typeof SomeSchema>;
```

### 3. Use immutable snapshots

Core objects should represent immutable facts whenever possible.

Examples:

```text
PullRequestSnapshot
CodeIndexVersion
IndexManifest
ContextBundle
ReviewRun artifact refs
CandidateFinding
PublishedFinding
UsageEvent
LLMCall
```

If a PR changes, create a new snapshot for the new `headSha`. Do not mutate the old snapshot.

### 4. Provider-neutral core, provider-specific adapters

The product starts with GitHub, but the contracts should not make GitHub concepts leak everywhere.

Good:

```ts
provider: "github"
providerRepoId: string
pullRequestNumber: number
providerMetadata?: Record<string, unknown>
```

Bad:

```ts
githubNodeId: string
githubInstallationToken: string
githubDiffPosition: number
```

GitHub-specific details should usually live in:

```text
/packages/github
/packages/publisher
```

The core contracts may include optional provider metadata, but the review engine should operate on normalized shapes.

### 5. Version anything that can cross process boundaries

Version these objects:

```text
IndexManifest
IndexRecord
JobEnvelope
PullRequestSnapshot
ContextBundle
Finding schemas
API DTOs if public
```

Versioning prevents future indexer/reviewer changes from becoming destructive migrations.

### 6. Stable IDs are critical

Use stable, prefixed IDs.

Two broad classes:

1. **Entity IDs** generated by the application, usually ULID-style.
2. **Content IDs** derived from deterministic hashes.

Examples:

```text
org_01HV...
usr_01HV...
repo_01HV...
rrn_01HV...

file_sha256...
sym_sha256...
chunk_sha256...
edge_sha256...
```

Content-derived IDs help compare outputs from different indexers and avoid database-generated coupling.

### 7. Contracts should not perform business logic

The contracts package should contain:

```text
schemas
types
validators
constants
enum definitions
pure helpers
test fixtures
```

It should not contain:

```text
database queries
GitHub API calls
LLM calls
queue clients
embedding calls
network requests
filesystem repo indexing
```

### 8. Metadata is okay, but typed fields come first

Use typed fields for things you actively depend on.

Use metadata only for extensibility:

```ts
metadata: Type.Optional(Type.Record(Type.String(), Type.Unknown()))
```

Do not hide important business logic in `metadata`.

### 9. Strict by default

For most objects, set `additionalProperties: false`.

Use explicit `metadata` fields for future extension.

This helps catch errors early, especially in LLM outputs and index artifacts.

### 10. Prefer additive changes

Do not casually remove or rename fields. Prefer:

- Add optional field.
- Add new enum value carefully.
- Add new record type.
- Add new versioned schema.
- Keep old validators during migration.

## Recommended technology choice

### Canonical schemas: TypeBox

Use:

```text
@sinclair/typebox
@sinclair/typebox/value
ajv
ajv-formats
```

Why TypeBox fits this project:

- Works well with Elysia.
- Produces JSON Schema.
- Fast when compiled with Ajv.
- TypeScript types come from schemas.
- Good for API/job/artifact contracts.
- JSON Schema can be used for OpenAI structured outputs, validation, and docs.

### Optional schema adapter: Zod

Some libraries, especially LLM tooling, support Zod ergonomically. Do not block that.

Recommended approach:

```text
contracts package: TypeBox canonical schemas
llm-gateway package: schema adapter or local Zod wrappers where needed
```

If you later decide Zod is preferable, the package boundary still works. The key is to keep one canonical schema source per contract.

## Global conventions

### Naming

Use schema/type pairs:

```ts
export const PullRequestSnapshotSchema = Type.Object({ ... });
export type PullRequestSnapshot = Static<typeof PullRequestSnapshotSchema>;
```

For enums:

```ts
export const ReviewStatusSchema = Type.Union([
  Type.Literal("queued"),
  Type.Literal("running"),
  Type.Literal("completed"),
  Type.Literal("failed")
]);

export type ReviewStatus = Static<typeof ReviewStatusSchema>;
```

For constants:

```ts
export const REVIEW_STATUS = {
  Queued: "queued",
  Running: "running",
  Completed: "completed",
  Failed: "failed"
} as const;
```

Do not use TypeScript `enum` unless you have a strong reason. Literal unions are easier to serialize and validate.

### Field casing

Use `camelCase` everywhere in contracts.

Examples:

```text
repoId
commitSha
createdAt
baseSha
headSha
pullRequestNumber
```

Database columns can use snake_case separately in Drizzle.

### Dates

Use ISO 8601 strings in contracts.

```ts
export const IsoDateTimeSchema = Type.String({
  format: "date-time"
});
```

Avoid passing `Date` objects across boundaries. Convert at the edges.

### Money and cost

Use integer micros for model costs and usage costs.

```ts
costMicros: number
```

One dollar is:

```text
1_000_000 micros
```

Avoid floats for billing.

### Paths

Repository paths should be normalized:

```text
src/auth/session.ts
```

Never:

```text
./src/auth/session.ts
/absolute/path/src/auth/session.ts
src\auth\session.ts
```

Define a `RepoPathSchema`.

### Unknown metadata

Use:

```ts
metadata?: Record<string, unknown>
```

But never store secrets in metadata.

### Null versus undefined

For JSON contracts, prefer optional fields over `null`.

Use `null` only when it has distinct semantic meaning.

### Error codes

Use string codes, not only messages.

Examples:

```text
contract.validation_failed
github.rate_limited
indexer.timeout
review.low_confidence
publisher.line_not_in_diff
```

### Traceability

Most persisted artifacts should have:

```ts
createdAt
traceId?
schemaVersion?
```

This makes review debugging much easier.

## Pull request snapshot contracts

The PR snapshot is a key object. It should represent the full normalized state needed for review at a specific moment.

### PullRequestSnapshot

```ts
export const PullRequestSnapshotSchema = Type.Object({
  snapshotId: Type.String({ pattern: "^prs_[A-Za-z0-9_-]+$" }),
  schemaVersion: Type.Literal("pull_request_snapshot.v1"),

  provider: GitProviderSchema,
  repoId: RepoIdSchema,
  installationId: InstallationIdSchema,

  providerRepoId: Type.String(),
  providerPullRequestId: Type.String(),
  pullRequestNumber: Type.Integer({ minimum: 1 }),

  title: Type.String(),
  body: Type.Optional(Type.String()),
```

### Index artifact re-exports

`@repo/index-schema` owns the concrete `IndexManifest`, `IndexRecord`, and record schemas. This package should not define local copies.

The contracts package may re-export the public artifact surface:

```ts
export type {
  IndexManifest,

  FileRecord,
  SymbolRecord,

  ChunkRecord
```

The canonical version strings are defined in #10:

```text
IndexManifest.schemaVersion = "index_artifact.v1"
IndexRecord.schemaVersion = "index_record.v1"
```

The concrete record schemas live in #10. Keep all artifact examples there so the importer, indexer, and contracts package do not drift.

```ts
import type { Static, TSchema } from "@sinclair/typebox";
import { ajv } from "./ajv";

export function parseWithSchema<T extends TSchema>(
  schemaName: string,
  schema: T,
  input: unknown
): Static<T> {
  const validate = ajv.compile(schema);
  const valid = validate(input);

  if (!valid) {
    throw {
      code: "contract.validation_failed",
      message: `Input failed schema validation: ${schemaName}`,
      schemaName,
      issues: (validate.errors ?? []).map((err) => ({
        path: err.instancePath || "/",
        message: err.message ?? "Invalid value",
        keyword: err.keyword
      }))
    };
  }

  return input as Static<T>;
}

export function safeParseWithSchema<T extends TSchema>(
  schemaName: string,
  schema: T,
  input: unknown
): { ok: true; value: Static<T> } | { ok: false; error: unknown } {
  try {
    return { ok: true, value: parseWithSchema(schemaName, schema, input) };
  } catch (error) {
    return { ok: false, error };
  }
}
```

Optimization note:

Compile validators once for hot paths. Do not compile schemas inside tight loops like JSONL artifact import.

Better pattern:

```ts
const validateIndexRecord = ajv.compile(IndexRecordSchema);

for await (const record of readJsonLines(path)) {
  if (!validateIndexRecord(record)) {
    // handle error
  }
}
```

## Schema versioning policy

Use explicit schema version strings:

```text
pull_request_snapshot.v1
index_artifact.v1
index_record.v1
context_bundle.v1
candidate_finding.v1
review_run.v1
```

When to bump major version:

- Rename required fields.
- Remove fields.
- Change semantic meaning.
- Tighten validation in a way that rejects previously valid data.
- Change ID generation semantics.
- Change record format.

When not to bump major version:

- Add optional fields.
- Add metadata.
- Add a new record type that old consumers can ignore.
- Add a new enum value only if consumers are designed to handle unknowns.

For index records, prefer adding new record types over bloating one record:

```text
route
test_mapping
dependency
framework_mapping
```

## Common mistakes to avoid

### Mistake 1: Contracts depend on implementation packages

Bad:

```ts
import { db } from "@repo/db";
```

Contracts should never import DB code.

### Mistake 2: Review engine consumes GitHub raw payloads

Bad:

```ts
reviewPullRequest(githubWebhookPayload)
```

Good:

```ts
reviewPullRequest(PullRequestSnapshot)
```

### Mistake 3: Indexer writes directly to the database

Bad:

```text
Indexer -> DB tables
```

Good:

```text
Indexer -> IndexArtifact -> Importer -> DB tables
```

### Mistake 4: LLM outputs become comments directly

Bad:

```text
LLM prose -> GitHub comment
```

Good:

```text
LLM output -> CandidateFinding -> validation -> ranking -> PublishedFinding
```

### Mistake 5: No artifact versions

If you do not version index records, swapping the indexer later becomes painful.

### Mistake 6: Too much optionality

Optional fields are useful, but if everything is optional, contracts become meaningless.

Required fields should represent the minimum object needed by consumers.

### Mistake 7: IDs generated by the database for content records

Database-generated IDs make it harder to compare indexer outputs.

Use deterministic IDs for files, symbols, chunks, and edges.

### Mistake 8: Storing secrets in metadata

Do not allow tokens, private keys, or raw authorization headers in metadata.
